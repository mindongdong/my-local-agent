# 0004: Manager LLM으로 qwen3.5:9b 채택, Gemma 3n E4B는 fallback

## Context

원본 spec section 1은 [[Manager]] LLM으로 **Gemma 3n E4B** (`gemma3n:e4b`, ~2GB)를 제안했다. 선택 근거는 메모리 경량성이었다.

그러나 프로젝트 요구 사항을 검토하면서 두 가지 약점이 드러났다.

1. **한국어 의도 파싱 품질**: Gemma 3n E4B는 영어 중심 모델로, 한국어 이슈 제목·본문의 의도를 파악하는 정확도가 낮다. [[Manager]]의 핵심 역할 중 하나가 한국어 이슈를 해석해 [[Worker]] 지시를 작성하는 것이므로, 의도 파싱 오류는 전체 파이프라인 품질에 직접 영향을 준다.

2. **JSON 도구 호출 일관성**: [[interrupt #1]], [[interrupt #2]], [[interrupt #3]] 처리를 위해 [[Manager]]는 구조화된 JSON 응답을 안정적으로 생성해야 한다. 경량 모델일수록 few-shot 없이 JSON schema를 준수하는 능력이 흔들리는 경향이 있다.

Ollama 라이브러리에서 qwen3 계열 후보를 확인한 결과, `qwen3:9b` 태그는 존재하지 않으며 실제로 선택 가능한 후보는 다음과 같다.

| 모델 | 크기 | 특이사항 |
|---|---|---|
| `qwen3:8b` | ~5.2GB | 경량, 짧은 context window |
| `qwen3.5:9b` (Q4_K_M) | ~6.6GB | 256K context window, multimodal |
| `qwen3:14b` | ~9.3GB | capability 우수, 메모리 부담 큼 |

Mac Mini M4 16GB 환경에서 OMC + cmux + FastAPI + Discord 봇이 동시 가동되는 조건을 고려하면, 각 후보의 메모리-성능 균형을 신중하게 판단해야 한다.

## Decision

**[[Manager]] LLM으로 `qwen3.5:9b` (Q4_K_M 기본 quant, 6.6GB)를 채택한다.**

**[[Gemma 3n E4B]]는 fallback으로 유지한다.** 메모리 압박이 임계값을 초과하거나 [[W0.5]] 검증에서 qwen3.5:9b의 capability가 충분하다고 확인되기 전까지는 기본 선택지를 qwen3.5:9b로 고정한다.

Pull 명령어: `ollama pull qwen3.5:9b`

채택 근거는 다음과 같다.

- **256K context window**: interrupt-heavy 워크플로우, 긴 이슈 본문, wiki retrieve 결과를 Manager 컨텍스트에 붙일 때 truncation 없이 처리 가능하다.
- **multimodal (text + image)**: 스크린샷·다이어그램 첨부 이슈를 향후 처리할 여지를 열어 둔다. W0~W1에서 즉시 필요하지 않더라도, 모델 교체 없이 기능을 추가할 수 있다.
- **한국어 + JSON 도구 호출 품질**: qwen 계열은 한국어 이해도와 structured output 일관성에서 Gemma E4B 대비 우위가 보고되어 있다.
- **Q4_K_M quant**: 6.6GB로 메모리 효율과 추론 품질의 균형을 잡는 기본 quant 선택이다.

## Consequences

**긍정적 영향**

- 한국어 이슈 의도 파싱 / JSON 도구 호출 정확도 향상 (vs [[Gemma 3n E4B]])
- 256K context window — [[interrupt #1]] 왕복 + 긴 이슈 본문 + wiki retrieve 결과 첨부에 모두 여유 있게 대응
- multimodal — 이미지 입력 파이프라인을 추후 모델 교체 없이 추가 가능
- Q4_K_M 기본 quant — Ollama 기본 pull과 동일, 별도 quant 설정 불필요

**부정적 영향 / 리스크**

- **메모리 점유 6.6GB**: Mac Mini 16GB 기준 baseline OS + OMC + FastAPI + Discord 봇 합산 시 여유 buffer ~2GB 내외로 빡빡하다. [[일일 토큰 cap]] 가드레일과 별개로, 동시 작업 캡을 1~2로 제한해야 할 수 있다.
  - **[2026-05-30 측정 정정]**: 6.6GB는 *디스크* 크기였고, 로드 시 **RAM 점유는 8.5GB** (ctx 4096 기준, `ollama ps`). 256K context는 명목 스펙으로, 16GB에서 full로 채우면 KV 캐시가 초과하므로 **Manager 컨텍스트를 보수적으로 캡**해야 한다. 모델 + Claude 세션 1개만으로 이미 free 105M + swapout 이력 → 메모리가 binding constraint임이 실측 확정. 상세: [[w0.5-validation]].
- **W0.5 검증 필수**: 도구 호출 / JSON schema 준수율을 실측해야 한다. 흔들리면 system prompt 강화 + JSON schema 강제로 보완하고, 그래도 부족하면 → [[ADR 0007]] 에서 `qwen3:14b` 전환 검토 (동시성 1로 축소).
  - **[2026-05-30 측정 — PASS]**: prompt에 JSON 구조 명시 + 코드 검증(temp=0) 방식으로 14개 한국어 케이스에서 **구조 14/14(100%), 의미 13/14(93%)**, 결정성 양호. 9B의 JSON 능력은 충분. **단 "JSON schema 강제"의 메커니즘 정정**: ollama `format`(스키마 객체)은 이 setup(ollama 0.24.0 + qwen3.5:9b)에서 강제되지 않음(probe: enum 스키마 무시) → 강제는 **프롬프트 구조 명시 + 코드측 파싱/검증 + 재시도**로 구현해야 한다. 상세: [[w0.5-validation]].
- **되돌리기**: [[Gemma 3n E4B]] fallback이 준비되어 있어 롤백 비용은 낮다. 단 W1 이후 Manager 프롬프트가 qwen3.5:9b 특성에 맞게 튜닝되면 교체 비용이 상승한다.

**되돌리기 난이도**: 낮음 (W0.5 이전) → 중간 (W1 이후 프롬프트 의존 증가)

## Alternatives considered

**(A) Gemma 3n E4B 유지 (원본 spec)**

메모리 ~2GB로 가장 가볍다. 그러나 한국어 의도 파싱 약점과 JSON 도구 호출 불안정성이 [[interrupt #1]]~[[interrupt #3]] 흐름에서 오류를 유발할 가능성이 높다. 채택하지 않는 이유: [[Manager]]의 핵심 책임이 한국어 파싱과 구조화 응답 생성인데, 해당 영역에서 capability 부족이 명확하다.

**(B) qwen3:8b (5.2GB)**

qwen3.5:9b 대비 1.4GB 가볍고 Ollama에서 검증된 태그가 존재한다. 그러나 context window가 짧고 multimodal 지원이 없다. qwen3.5:9b와 메모리 차이가 1.4GB인 반면 capability 손실이 있어 가성비가 낮다고 판단. 채택하지 않는 이유: 동일 계열에서 qwen3.5:9b가 더 나은 선택이다.

**(C) qwen3:14b (9.3GB)**

capability는 가장 우수하지만 9.3GB 점유로 16GB Mac Mini에서 OMC + cmux + FastAPI + Discord 동시 가동 시 스왑이 거의 확실하다. [[작업당 turn cap]] 가드레일 적용 시에도 latency 급등 위험이 크다. 채택하지 않는 이유: 메모리 초과 리스크가 실용성을 제한한다. W0.5 측정 결과 qwen3.5:9b capability 부족이 확인되면 [[ADR 0007]]에서 재검토.

**(D) 외부 API (Claude Haiku 등)**

로컬 자율성 목표에 반한다. 비용 발생 + 인터넷 의존성 + 한국어 이슈 내용 외부 전송 리스크. 채택하지 않는 이유: 프로젝트 핵심 목표가 로컬 자율 에이전트이므로 처음부터 배제.
