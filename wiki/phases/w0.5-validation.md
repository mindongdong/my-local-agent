### 메모리 풋프린트 — ⚠️ plan 가정과 상이

| 항목 | plan §4 가정 | 실측 (2026-05-30) | 측정 방법 |
|---|---|---|---|
| qwen3.5:9b 로드 점유 | ~6.6GB | **8.5GB** (6.6GB는 *디스크* 크기) | `ollama ps` (ctx 4096, 100% GPU) |
| 로드 컨텍스트 | 256K 활용 가정 | 기본 **4096**으로 로드 | `ollama ps` CONTEXT 열 |
| 시스템 메모리 | buffer ~2GB | **PhysMem 15G used / 105M free**, swapout 이력 존재 | `top -l 1` (모델 + Claude 세션 1개 상태) |
| 추론 속도 | (없음) | ~25 tok/s (warm) | `/api/generate` eval_count/eval_duration |

**해석**
- 8.5GB는 ctx 4096 기준. 256K 컨텍스트를 실제로 채우면 KV 캐시가 16GB를 초과 → **256K는 명목 스펙일 뿐, Manager는 컨텍스트를 보수적으로 캡해야 함**.
- 모델 + Claude 세션 1개만으로 이미 free 105M. macOS 메모리 회계상 "used"에 회수 가능한 캐시가 포함되나, swapout 이력은 실제 압박 신호.
- → 메모리가 시스템의 **binding constraint**임이 실측으로 확정 ([work-plan.md](../../work-plan.md) §10 High 리스크 = 현실).

### cmux round-trip 통신 — ✅ 통과 (≥99% 기준)

| 항목 | 값 |
|---|---|
| 표본 | **300회** (run1 100 + run2 200, 동일 probe surface 재사용) |
| 성공 | **300/300 (100%)** |
| 실패율 95% 신뢰 상한 | ~1% (rule of three: 3/300) |
| 평균 round-trip 지연 | 0.277s |
| 최대 지연 | 0.329s |
| 첫 시도(0.2s) 캡처율 | run2 전부 (200/200) |

방식 (cmux 0.64.10 **실제 API** — 이 세션이 cmux `workspace:2/surface:4` 안에서 직접 구동):
- **caller 파악**: `cmux identify --json` → Manager가 자기 위치 인지
- **spawn**: `cmux new-workspace --name w0.5-probe --focus false` → `workspace:5` (포커스 안 뺏음)
- **surface ref**: `cmux list-pane-surfaces --workspace workspace:5` → `surface:9`
- **입력**: `cmux send --surface <s> "<cmd>"` + `cmux send-key --surface <s> Enter` (텍스트/키 분리)
- **캡처**: `cmux read-screen --surface <s>` (tmux-compat `capture-pane`도 동작)
- **정리**: `cmux close-workspace` → 토폴로지 원상복구, caller 포커스(workspace:2) 유지 = **비파괴 자동화 확인**
- **성공 판정**: 출력 토큰을 `$((i*7))` 산술 결과로 설계 → "타이핑됨"이 아니라 "실제 실행됨"만 카운트

판정: **cmux 통신 안정성 PASS** → [[ADR 0002]] 핵심 리스크([work-plan.md](../../work-plan.md) §10 Medium) 해소.
단 이 측정은 **셸 echo 프리미티브** 기준 — OMC/claude TUI Worker 구동 시 안정성은 `autopilot:` 단발 호출 항목에서 별도 검증.

> **API 어휘 정정**: plan/ADR이 가정한 tmux식 중 `send-keys`(복수형)·`new-session`은 cmux 0.64.10에 **없음**. 정정 대상 = `send-keys`→`send`+`send-key`, `new-session -d`→`new-workspace --focus false`. `capture-pane`은 tmux-compat로 유효하므로 유지 가능. (wiki 전반 정합화는 아래 TODO)

### `ulw`/동시 워커 메모리 — ✅ 측정 (워커 수는 메모리 리스크 아님)

| 상태 | free% | swap used | 비고 |
|---|---|---|---|
| 모델 언로드 (base) | 76% | 4.26GB | macOS + Chrome + VSCode + claude×3 + cmux |
| **모델 로드** (+8.5GB) | **14~17%** | 5.54GB (**+1.3GB**) | ollama runner RSS 7.1GB. 지배적 메모리 이벤트 |
| **+2 claude 워커** | 17% (**변동 없음**) | ~동일 | 신규 워커 각 RSS **~213MB**, 합 +0.43GB |
| 워커 정리 후 | 18% | — | `close-workspace`로 즉시 회수 |

방법: 모델 `keep_alive:"5m"`로 pin → `cmux new-workspace --command 'claude' --focus false` ×2 spawn → 14s 후 peak 측정 → `close-workspace` ×2. 측정 내내 기존 claude 3개(VSCode 확장·resume 세션·이 세션) 가동 = 사실상 **≥3 동시 워커** 상태.

**판정 / 핵심 통찰**:
- **`ulw` 병렬은 서버측 subagent 실행** → 로컬 RAM을 N배로 늘리지 않는다. plan §4의 "ulw 5병렬 → +1.5GB" 가정은 **반증됨**.
- 로컬 메모리의 진짜 레버 = **Manager 모델(8.5GB) + 데스크톱 GUI 앱(Chrome/VSCode ~1~1.5GB)**, 워커 수가 아님 (워커 ~213MB/개).
- 2 동시 워커는 메모리상 **안전** (free% 평탄). 단 이 머신은 모델+데스크톱만으로 이미 스왑 중 → **헤드리스 Mac Mini(plan 권장)에서 Chrome/VSCode 미가동 시 ~1~1.5GB 회수**가 14GB 임계값 준수의 관건.
- 완화책: 헤드리스 운영, Manager 컨텍스트 캡, `keep_alive` 튜닝(유휴 시 모델 언로드).

> 한계: 워커 RSS는 **idle-at-load** 기준 (실작업 시 Node heap 소폭 증가). 측정은 Chrome/VSCode가 있는 **개발 데스크톱** 기준 — 헤드리스 타겟과 다름. 리터럴 `ulw` 실행은 안 함 (서버측이라 로컬 메모리 신호가 동일 → 토큰 낭비 회피).

### JSON schema 준수율 — ✅ 측정 (prompt-explicit 방식, plan §10 High)

먼저 ollama `format`(JSON schema 객체) 강제를 시도했으나 **강제되지 않음**을 발견:
- **probe**: `{verdict: enum[APPROVE,REJECT]}` 스키마 + "Say hello" 프롬프트 → 출력 `"Hello there! It's so nice to meet you."` (JSON도 아님, 스키마 완전 무시)
- triage 스키마 강제 시 모델이 내 키(`task_type/priority/needs_clarification`)를 무시하고 `issue_type/severity/category`를 임의 생성
- → **ollama 0.24.0 + qwen3.5:9b + `/api/chat` 조합에서 `format` 스키마가 constrained decoding으로 작동하지 않음.** Manager는 `format`에 의존하면 안 됨. (ollama 버전/모델 arch 업그레이드 시 재확인 가치 있음)

올바른 방식 = **프롬프트에 정확한 JSON 구조 명시 + 코드측 파싱/검증** (temp=0). 14개 한국어 이슈 케이스:

| 작업 | 구조 유효성 | 의미 정확도 |
|---|---|---|
| 이슈 triage (`task_type`/`priority`/`needs_clarification`) | **8/8 (100%)** | 7/8 (88%) |
| 정보 충분성 (`sufficient`/`missing_info`/`summary_ko`) | **6/6 (100%)** | 6/6 (100%) |
| **합계** | **14/14 (100%)** | **13/14 (93%)** |

- 유일한 의미 오류: "회원 탈퇴 기능 어떻게 구현?" → `task_type=question`(정답)이나 `needs_clarification=True`(기대 False). 경계 케이스이고 type은 정답.
- **결정성**: temp=0 3회 반복 전부 동일(uniq=1). 기본 temp=1에서도 명확한 케이스는 5회 전부 동일 → Manager는 temp=0 권장하되 분류 안정성 양호.

판정: **JSON 신뢰성 PASS** (구조 100%, 의미 93%) → [work-plan.md](../../work-plan.md) §10 High 리스크 완화. 단 완화책은 "ollama format 강제"가 아니라 **"프롬프트에 구조 명시 + 코드측 검증 + (파싱 실패 시) 재시도"**로 정정.

> 측정 함정 기록: 1차 측정은 ollama `format`에 의존 → triage 0/8 → 잘못된 "21%"가 나왔다. format이 강제되지 않아 모델이 임의 키를 생성한 게 원인. raw 출력 확인으로 규명 후 방법 교정 → 93%. **측정 도구 가정을 먼저 검증할 것.**

## Mac Mini 재측정 (2026-05-30)

타겟 **Mac Mini (M4, 16GB = 17,179,869,184 bytes, macOS 26.3)** 에서 W0.5 핵심 3 Go 기준을 헤드리스로 재현 ([[w1-handoff]] §1·§6 요구). 위 "## 측정 데이터"는 개발용 다른 Mac 기준이며, 메모리는 머신 특정값이므로 16GB 타겟에서 다시 측정했다.

**측정 위생** (이전 세션 도구 출력 신뢰성 문제 대응): 모든 측정은 **단일 명령 → 결과 파일 → Read 교차검증**, 대량 병렬 Bash 지양, raw 출력만 기록(집계값 신뢰 안 함).

**헤드리스 셋업**: GUI 앱 종료 — VSCode(Code)·Discord·KakaoTalk·iTerm2·Preview (정상 종료, 잔여 헬퍼 0 확인). 호스트는 **cmux.app(ghostty 백엔드)** 라 VSCode와 무관. Chrome 미가동. 잔존 GUI = cmux(호스트)·Finder.

### ① 메모리 — ⚠️ 조건부 PASS (메트릭 주석)

| 상태 | top PhysMem used | unused | swap used | pressure(free%) | 비고 |
|---|---|---|---|---|---|
| 헤드리스 baseline (모델 언로드) | **8,494M** | 7,346M | 113.5M | — | macOS + cmux + 이 claude 세션 |
| 모델 로드 (ctx 4096, 활성+idle) | **15G** | 130~256M | **113.5M** | 28~29% | 모델 runner RSS **7.06 GiB**, ollama SIZE 8.5GB, 13.3 tok/s(600tok warm) |
| **모델 + 2 신규 claude 워커 (3 동시, peak)** | **15G** | 189M | **113.5M (고정)** | 27% | 워커 ~375MB/개, compressor +148M로 흡수 |
| 워커 종료 후 | 15G | 757M | 113.5M | 27% | 즉시 회수 |

- **리터럴 기준 "peak < 14GB"는 초과** (15G used). 그러나 **swap이 baseline부터 모델+3워커까지 113.5M로 전혀 안 늘고**, memory_pressure도 normal(warn/critical 없음) 유지 → **실제 메모리 압박/스왑 스래싱 없음**.
- **메트릭 주석 (핵심)**: Apple Silicon **통합메모리**에서 GPU(Metal) 상주 모델 가중치(~7.4GB)가 **wired**로 잡혀 `top`의 "used"가 모델 로드만으로 ~15G가 된다(wired 2,067M→9,558M). 따라서 **`top used`는 16GB 머신에서 모델을 띄우면 항상 14GB를 넘으며, 압박의 올바른 게이지가 아니다.** 실효 게이지 = **swap 증가량 + pressure level**, 둘 다 건강.
- **GUI를 다 닫아도 14GB 밑으로 안 내려간다**: 병목은 GUI가 아니라 **모델 runner 7.4GB + macOS baseline ~8GB**. 에이전트 스택(claude 363M + cmux 128M + node hud/mcp)은 합 ~0.5GiB로 미미. → 헤드리스는 도움이 되나 바닥은 모델+OS다. **메모리의 진짜 레버 = keep_alive(유휴 시 모델 언로드 → 7.4GB 회수)**, 단순 GUI 종료가 아님.
- **판정**: 기준 ① 문구("< 14GB")는 유지하되, Mac Mini는 **"스왑 무증가 & pressure normal" = 기준의 의도(스왑 스래싱 없이 동작) 충족**으로 **조건부 PASS**. 헤드룸은 얇음(unused ~170M) → 동시성/추가 컴포넌트(Discord bot·FastAPI·SQLite)는 운영 중 재측정 필요.

### ② cmux round-trip — ✅ PASS (300/300, 100%)

| 항목 | 값 |
|---|---|
| 표본 / 성공 | **300 / 300 (100%)**, 실패 0 |
| first-try 캡처(0.2s 폴) | **300/300** |
| 평균 trial | ~0.25s (i=50마다 12~13s/50trial) |

방식: 이 세션이 cmux `workspace:2/surface:2`에서 구동(`cmux identify`). probe workspace spawn → `send "echo Z<i>_<i*7>Z\n"` → `read-screen`에서 `^Z<i>_<i*7>Z$` 앵커 매칭으로 **"실제 실행"만 카운트**(산술 출력). 종료 후 `close-workspace` → caller workspace:2 포커스 유지 = **비파괴 확인**.

> **cmux 0.64.10 API 보완 (Mac Mini에서 드러난 정정 — [[w1-handoff]]에 반영 필요)**:
> 1. **`--surface <ref>` 단독은 "Surface is not a terminal" 오류** → **`--window window:1` 컨텍스트를 함께** 줘야 함 (`surface:N` 짧은 ref는 window-scoped index라 모호; UUID 단독도 실패). 작동형: `cmux send --surface surface:N --window window:1 "..."`.
> 2. **기본 `new-workspace`는 비-터미널 surface**(welcome) 생성 → 터미널은 **`--layout '{"pane":{"surfaces":[{"type":"terminal","command":"zsh"}]}}'`** 로 명시 spawn.
> 3. `send`는 `"...\n"` 이스케이프로 텍스트+Enter 1콜 가능. `send-key`는 `enter`(소문자).
>
> 원본(다른 머신) 측정은 `--surface`만으로 통했으나(단일 window 추정), 타겟 Mac Mini는 window 컨텍스트 필수. W1 cmux 래퍼는 **항상 `--window` 포함 + 터미널 layout spawn**으로 구현한다.

### ③ ulw / 동시 워커 — ✅ PASS

위 ① 표의 "모델 + 2 신규 워커" 행. claude 워커 2개(+~0.75GB) 추가에도 **swap 고정·pressure normal**, macOS 압축으로 흡수. 원본의 "워커 수는 메모리 리스크 아님(서버측 ulw)" 결론을 Mac Mini에서 재현. close 시 즉시 회수.

### Mac Mini 종합 판정

**3 Go 기준 모두 Mac Mini 16GB 헤드리스에서 재현 — ②③ PASS, ① 조건부 PASS(메트릭 주석).** [[ADR 0007]] **Go (헤드리스 전제) 유지**, Mac Mini addendum 추가. 운영 전제에 **keep_alive 단명(버스트 로드)** 명시(모델은 Manager 추론 버스트에만 로드, 유휴 시 언로드로 7.4GB 회수). 모델 라우팅(`eco:`→haiku)은 여전히 [[W1]]로 연기.

## 흔들렸던 가정

1. **6.6GB = 메모리 점유** (plan §4, ADR 0004) → 실제로는 *디스크* 크기. **로드 시 RAM은 8.5GB@4k ctx**, 컨텍스트 키우면 더 증가.
2. **256K context를 자유롭게 활용** (ADR 0004 채택 근거) → 16GB에서는 full context 불가. Manager 컨텍스트 캡 설정이 필수.
3. **`qwen3.5`가 mislabeled/Gemma 파생 아닌가** (검증 중 일시 의심) → 메타데이터 + clean 프로브로 **정품 Qwen3.5 (Tongyi Lab)** 확정. thinking 트레이스 헛소리에 속지 말 것.
4. **`ulw` 병렬이 로컬 메모리를 N배로 늘린다** (plan §4 "ulw 5병렬 +1.5GB") → **반증**. ulw subagent는 서버측 실행이라 로컬 RAM 영향 미미. 메모리 리스크는 워커 수가 아니라 **모델 8.5GB + 데스크톱 앱**. 워커는 ~213MB/개.
5. **ollama `format`(JSON schema)이 출력을 강제한다** (plan §10 "JSON schema 강제" 완화책의 암묵 전제) → **반증**. 이 setup에서 format은 스키마는커녕 JSON조차 강제하지 않음. 9B의 JSON 능력은 충분(93%)하나, 강제 메커니즘은 ollama format이 아니라 **프롬프트 구조 명시 + 코드 검증**이어야 한다. 측정 도구 가정을 먼저 검증하지 않으면 "21%" 같은 오판이 나온다.
