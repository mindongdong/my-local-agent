## 1. 목표

W1은 **하드코딩 영문 prompt** → cmux 워커였다. W2는 그 앞단에 Manager 두뇌를 붙여,
입력을 **실제 GitHub 이슈**로 바꾼다.

```
GitHub 이슈(#n)
  → [parse_issue]        gh fetch + 한국어 의도 파싱 (qwen)         ┐
  → [assess_sufficiency] 정보 충분성 JSON 판단 (qwen, 코드측 검증)  │ W2 신규
  → [build_prompt]       영문 작업 명세 + magic keyword 라우팅 (qwen) ┘
  → [W1 척추] cmux claude 워커 → git diff → qwen 한국어 요약          (재사용)
  → stdout (Discord 자리)
```

## 3. 핵심 결정 (사용자 승인)

### 3.1 LangGraph는 W3로 연기 (의존성 0)

work-plan §9는 "LangGraph minimal"을 명시하지만, **W2 시점엔 분기·interrupt·
checkpointer가 전혀 없다**(그 셋은 W3 도입). 분기 없는 3노드 직렬에 LangGraph를
끼우면 이점 ≈ 0, 의존성만 늘어 [[YAGNI]]/Simplicity First 위반. 대신:

- 노드를 `State -> State` 순수 함수로 작성 + 얇은 순차 러너로 연결.
- 시그니처를 LangGraph 노드 형태에 맞춰 둠 → **W3는 동일 함수를 `StateGraph`에
  등록하고 `assess → interrupt#1 → build` 분기/체크포인트만 추가** (재작성 아님).

### 3.2 keyword는 선택·기록만 (호출은 차기)

`build_prompt`가 task_type으로 magic keyword를 고르지만(system-design §3 라우팅 표),
실제 OMC keyword **호출**은 별도 노드(`invoke_omc_autopilot`)이며 W2 조각이 아니다.
E2E는 W1에서 검증된 평문 `claude -p`에 en_prompt를 인서트하고, 고른 keyword는
추적용으로 로깅만 한다. (autopilot: 등 OMC 모드 실주입 + `/trace` 라우팅 관측은
OMC 본격 통합 시 — [[ADR 0007]] Addendum의 연기 항목과 연결.)

### 3.3 정보 부족 분기는 출력 후 중단 (interrupt는 W3)

`sufficient=false`면 `build_prompt`를 건너뛰고 한국어 질문 목록을 출력 후 중단(exit 2).
사용자에게 되묻고 응답을 받아 resume하는 interrupt#1 루프는 W3.

## 4. 검증 (2026-06-01)

테스트 이슈 [#9](https://github.com/mindongdong/my-local-agent/issues/9)
(영어 제목 + 한국어 본문 → 한국어 의도 파싱 + 한→영 prompt 빌드 동시 실증, 검증 후 close):

```
[parse_issue] #9 'Add an ocean haiku to NOTES.md'  task_type=feature  힌트=NOTES.md
[assess_sufficiency] sufficient=True — 대상/위치/형식 명확
[build_prompt] keyword=autopilot:
   EN="Append a single English haiku about the ocean (exactly three lines) to the very end of NOTES.md. Do not create or modify any other files."
[worker] exit=0, 18.0s, diff 211B
[summarize] "NOTES.md 파일에 시적인 문장 3 줄을 추가했습니다"
```

순수 로직 단위 테스트 **7/7 통과**(키워드 라우팅, 상태 불변성, 이슈번호 파싱,
3개 노드 검증기, JSON 추출 견고성).

## 5. 흔들린 가정 / 관측

- **9B 단일 필드 오류는 파이프라인을 깨지 않음**: parse의 `summary_ko`가 "haiku 1 줄"로
  부정확(본문은 3줄)이었으나, `build_prompt`가 같은 이슈 본문을 다시 읽어 영문 명세를
  "exactly three lines"로 정확히 생성 → 워커 결과 정상. 노드마다 원본을 다시 참조하는
  구조가 9B의 요약 흔들림에 견딘다. (단, 의도 요약을 사용자에게 보여줄 땐 부정확 가능 —
  W4 승인 게이트에서 사용자가 교정 가능해야 함.)
- **gh fetch는 읽기 전용**: `gh issue view`만 사용 — L2 자율성(쓰기/PR은 Manager 승인 후).
