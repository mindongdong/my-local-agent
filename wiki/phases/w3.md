## 1. 목표

W2는 정보 부족 시 질문만 출력하고 중단(exit 2)했다. W3는 **되묻고 재개**한다:
W2가 연기한 LangGraph 전환 위에 interrupt#1 + SQLite checkpointer를 얹는다.

```
START → parse → assess ─┬─ sufficient ──→ build → END
                        └─ insufficient → clarify(interrupt#1)
                               ↑________________ 답변 → assess (재판단)
```

## 2. PR-a — interrupt 기계장치 (CLI, ✅ 완료 2026-06-03)

### 구현 (`spike/w3/`)

| 파일 | 역할 |
|---|---|
| `manager_graph_lg.py` | LangGraph 그래프(W2 3노드 재사용 래퍼 + `clarify` interrupt 노드 + 조건부 엣지) + `make_saver`(allowlist serde) |

### W2 노드 재사용 (수정 0)

`parse_issue`/`assess_sufficiency`/`build_prompt`를 그대로 노드 래퍼에 등록. 정보 부족
답변은 **이슈 본문에 Q&A append**(`append_clarification`)해서 assess 재실행 — assess/build가
더 풍부한 본문을 그대로 읽으므로 상태에 새 필드를 더하거나 W2 노드를 고칠 필요가 없다.
→ W2가 약속한 "**노드 등록만 + 분기/체크포인트 추가**"를 그대로 이행.

### serde 결정 (probe 선검증)

W2 dataclass(`ManagerState`/`Issue`/`Intent`/`Sufficiency`)를 그래프 state로 그대로 사용.
일회성 probe로 **중첩 frozen dataclass가 interrupt 경계에서 SQLite 직렬화/복원됨**을 확인.
단 미등록 커스텀 타입은 향후 버전에서 차단 경고 → `JsonPlusSerializer(allowed_msgpack_modules=
[("manager_state", …)])`로 등록해 future-proof(경고 제거 확인).

### 검증 (이슈 [#11](https://github.com/mindongdong/my-local-agent/issues/11), 모호하게 작성)

```
[parse] #11  intent=refactor
[assess] sufficient=False — 정리 기준/대상 누락
  ⏸ interrupt#1: 1. 어떤 기준으로 정리? 2. 유지/삭제 대상?
  [answer] 중복된 줄을 제거하고 제목 '# Notes'만 남겨주세요. 다른 파일은 건드리지 마세요.
[assess] sufficient=True  (clarify 1라운드)
[build] keyword=ralph:  EN="In NOTES.md, remove all duplicate lines and keep only the heading '# Notes'. …"
```

- interrupt/resume/checkpointer 동작 + **serde 경고 없음**(allowlist).
- 순수 로직 단위 테스트 **5/5 통과**.

### 미실행 — 워커 포함 전체 E2E (cmux 블록)

테스트 시점 **cmux 소켓이 broken-pipe**(`cmux ping` 3회 실패 — 소켓 파일 06-01자,
세션이 06-01~06-03 다일 구동되며 데몬 측 끊김; cmux 앱 재시작 필요). 워커 단계는
**W1-검증 경로**(이번 세션 W2 E2E에서 exit=0 정상)이며 W3 신규 코드가 아니다.
cmux 소켓 정상화 후 재실행. **PR-a 핵심(interrupt 기계장치)은 `--dry`로 완전 증명됨.**

## 3. PR-b — Discord 어댑터 (예정)

`spike/w1/discord_bot.py` 확장:
- 멘션/이슈 링크 → graph 시작(`thread_id` = 새 Discord thread.id).
- `insufficient` → interrupt payload의 한국어 질문을 **thread에 포스팅**하고 봇은 반환
  (W1의 fire-and-forget이 아니라, 일시정지 상태를 checkpointer가 보관).
- **thread reply 수신** → 그 `thread.id`로 checkpointer thread 찾아 `Command(resume=답변)`
  → 충분해지면 build → 워커 → 결과 포스팅.
- 핵심: **Discord thread.id ↔ LangGraph checkpointer thread_id 1:1 매핑** (SQLite 영속이라
  봇 재시작/대기 시간을 넘어 resume 가능).

## 4. 설계 메모 / W4로 넘기는 질문

- **라운드 캡**: `clarify`는 항상 일시정지 → 그래프 자체 무한루프 불가(매 라운드 = 호출측
  1 resume). 하니스 `MAX_ROUNDS=3` 캡. 실운영(Discord)은 계속 질문 또는 타임아웃 정책 필요.
- **parse 1회**: 의도(intent)는 원본 본문에서 1회 산출. 명료화는 assess/build가 본문 재참조로
  반영. intent 자체 갱신이 필요한 케이스가 보이면 W4에서 재parse 고려.
- **interrupt #2/#3 (승인 게이트)**: W4. 같은 interrupt/resume 기계장치를 `summarize_diff`
  후(코드 승인)·PR 생성 전(PR 승인)에 재사용. reaction(✅/❌) 매핑은 W4에서.
- **OMC `/deep-interview` ↔ interrupt#1**: [[ADR 0003]]가 지목한 별도 round-trip(워커가
  실행 중 되묻는 경우)은 OMC 본격 통합 시.
